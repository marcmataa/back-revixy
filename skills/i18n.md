# SKILL: I18N ARCHITECT (Internationalization for Backend Services)

## 1. ROLE & RESPONSIBILITY

You are a Senior Backend Engineer specialized in internationalization (i18n) for SaaS applications.
Your mission is to ensure that ALL user-facing text in REVIXY — emails, alerts, notifications —
respects the store's configured language (`store.language`).

**Scope of i18n in REVIXY:**

- ✅ Email subjects and bodies (`notification.service.js`)
- ✅ Alert titles and messages (`alertEngine.service.js`)
- ✅ AI system prompt language instruction (`contextBuilder.service.js`) — already implemented
- ❌ Server logs — always in English (never translate logs)
- ❌ Error messages thrown in services — always in English (for developer debugging)
- ❌ MongoDB field names or API response keys — always in English

---

## 2. ARCHITECTURE (MANDATORY)

### Single Source of Truth

All translations live in ONE file: `src/config/i18n.config.js`
Never hardcode translated strings anywhere else in the codebase.
Never use external i18n libraries (i18next, react-i18next, etc.) — they are frontend tools.

### Pattern

```
store.language → getT(store.language) → t.namespace.key or t.namespace.messageFunction()
```

### File Structure

```
src/
├── config/
│   └── i18n.config.js   ← single source of truth for all translations
└── services/
    ├── notification.service.js  ← uses getT() for email text
    └── alertEngine.service.js   ← uses getT() for alert titles/messages
```

---

## 3. `i18n.config.js` STRUCTURE (MANDATORY)

The file must export:

1. `translations` object — nested by language → namespace → key or function
2. `getT(language)` helper — returns the translation object with fallback to `"es"`

### Supported Languages

```js
const SUPPORTED_LANGUAGES = ["es", "en", "ca"];
const DEFAULT_LANGUAGE = "es";
```

### Translation Namespaces

Each language must have these namespaces:

**`alerts`** — Alert type titles (used as email subjects and alert titles):

```js
alerts: {
  ROAS_BELOW_BREAKEVEN: String,
  PROFIT_NEGATIVE: String,
  REVENUE_DROP: String,
  CTR_DECLINING: String,
  CPC_SPIKE: String,
  ROAS_DECLINING: String,
  SCALE_OPPORTUNITY: String,
  HIGH_MARGIN_DAY: String,
}
```

**`email`** — Static labels used in email templates:

```js
email: {
  open_dashboard: String,
  affected_metric: String,
  current_value: String,
  average_value: String,
  store_label: String,
  critical_alert_title: String,
  consecutive_days: String,
}
```

**`messages`** — Dynamic message functions (NEVER use string concatenation):

```js
messages: {
  // Siempre funciones — nunca concatenación de strings
  ROAS_BELOW_BREAKEVEN: (roas, breakEven, days) => String,
  PROFIT_NEGATIVE: (profitCents, days) => String,
  REVENUE_DROP: (revenueCents, averageCents, days) => String,
  CTR_DECLINING: (ctr, average) => String,
  CPC_SPIKE: (cpc, average) => String,
  SCALE_OPPORTUNITY: (roas, breakEven, adSpendCents, averageCents) => String,
}
```

### `getT` Helper (MANDATORY)

```js
// Devuelve las traducciones del idioma solicitado con fallback a español
export const getT = (language) => {
  const lang = SUPPORTED_LANGUAGES.includes(language)
    ? language
    : DEFAULT_LANGUAGE;
  return translations[lang] ?? translations[DEFAULT_LANGUAGE];
};
```

## 4. USAGE RULES (NON-NEGOTIABLE)

### Always use `getT`, never hardcode

```js
// ❌ WRONG — hardcoded English
const subject = "🚨 REVIXY Alert: ROAS below break-even threshold";

// ✅ CORRECT — respects store language
const t = getT(store?.language);
const subject = t.alerts.ROAS_BELOW_BREAKEVEN;
```

### Always use message functions, never string concatenation

```js
// ❌ WRONG — not translatable, breaks in other languages
const message = "ROAS is " + roas + " vs break-even " + breakEven;

// ✅ CORRECT — translatable, language-aware
const t = getT(store?.language);
const message = t.messages.ROAS_BELOW_BREAKEVEN(roas, breakEven, days);
```

### Alert type keys must match exactly

The keys in `t.alerts` and `t.messages` must match the `alert.type` values
from `alertEngine.service.js`:
`ROAS_BELOW_BREAKEVEN`, `PROFIT_NEGATIVE`, `REVENUE_DROP`, `CTR_DECLINING`,
`CPC_SPIKE`, `ROAS_DECLINING`, `SCALE_OPPORTUNITY`, `HIGH_MARGIN_DAY`.

Use this pattern to get the title dynamically:

```js
const t = getT(store?.language);
// Fallback al título original del alerta si no existe traducción
const alertTitle = t.alerts[alert.type] ?? alert.title;
const alertMessage = t.messages[alert.type]
  ? t.messages[alert.type](
      alert.currentValue,
      alert.averageValue,
      alert.consecutiveDays,
    )
  : alert.message;
```

### Always get `store.language` from the store document

```js
// En notification.service.js — siempre buscar el language del store en BD
const store = await Store.findById(storeId)
  .select("shopifyDomain language")
  .lean();
const t = getT(store?.language);
```

## 5. MONETARY VALUES IN MESSAGES

All monetary values from `DailyStats` and alerts are stored in CENTS.
Message functions that receive monetary values must convert them:

```js
// Las funciones de mensaje convierten céntimos a formato legible
PROFIT_NEGATIVE: (profitCents, days) => {
  const profit = (Number(profitCents) / 100).toFixed(2);
  return `El beneficio neto es ${profit}€ y ha estado negativo durante ${days} días.`;
},
```

For ratio fields (ROAS, CTR, CPC) — NOT cents, use directly:

```js
// ROAS, CTR y CPC son ratios — no dividir por 100
ROAS_BELOW_BREAKEVEN: (roas, breakEven, days) =>
  `El ROAS es ${Number(roas).toFixed(2)} vs punto de equilibrio ${Number(breakEven).toFixed(2)} durante ${days} días.`,
```

### Message Functions — Value Format Contract

The `messages` template functions expect values **already formatted as readable strings**.
The caller is responsible for converting cents to currency before passing to the function.

```js
// ✅ Correcto — formatear antes de llamar
const profitFormatted = (alert.currentValue / 100).toFixed(2) + " " + currency;
const body = t.messages.profit_negative(profitFormatted, alert.consecutiveDays);
// Result: "El beneficio neto es -5,25 EUR y ha estado negativo durante 2 días."

// ❌ Incorrecto — pasar céntimos directamente
const body = t.messages.profit_negative(
  alert.currentValue,
  alert.consecutiveDays,
);
// Result: "El beneficio neto es -525 y ha estado negativo durante 2 días." ← raro
```

**Fields that are in CENTS and need conversion before passing:**

- `alert.currentValue` — always in cents for monetary metrics
- `alert.averageValue` — always in cents for monetary metrics

**Fields that are NOT in cents (pass directly):**

- `alert.consecutiveDays` — integer, no conversion needed
- `alert.affectedMetric` — string, no conversion needed
- ROAS values — ratios, no conversion needed

## 6. EMAIL TEMPLATE I18N

In `notification.service.js`, the `buildHtmlEmail` function must accept `t` as parameter
and use it for ALL user-facing text:

```js
// Construimos el email con las traducciones del idioma de la tienda
function buildHtmlEmail({ storeName, alert, t }) {
  const alertTitle = t.alerts[alert.type] ?? alert.title;
  const alertMessage = t.messages[alert.type]
    ? t.messages[alert.type](
        alert.currentValue,
        alert.averageValue,
        alert.consecutiveDays,
      )
    : alert.message;

  return `
    <div style="font-family:Arial,sans-serif;max-width:600px;margin:0 auto;padding:16px;color:#111827;">
      <h2 style="margin-bottom:12px;">${t.email.critical_alert_title}</h2>
      <p><strong>${t.email.store_label}:</strong> ${storeName}</p>
      <p>${alertMessage}</p>
      <p><strong>${t.email.affected_metric}:</strong> ${alert.affectedMetric}</p>
      <p><strong>${t.email.current_value}:</strong> ${alert.currentValue}</p>
      <p><strong>${t.email.average_value}:</strong> ${alert.averageValue}</p>
      <a href="${process.env.FRONTEND_URL}/dashboard"
         style="display:inline-block;padding:10px 14px;background:#111827;color:#ffffff;text-decoration:none;border-radius:6px;">
        ${t.email.open_dashboard}
      </a>
    </div>
  `;
}
```

The `sendCriticalAlertEmail` function must:

1. Receive `store` (with `language` field) as parameter.
2. Call `getT(store?.language)` to get translations.
3. Pass `t` to `buildHtmlEmail`.
4. Use `t.alerts[alert.type]` as email subject.

## 7. ADDING NEW LANGUAGES

To add a new language (e.g. French `"fr"`):

1. Add `"fr"` to `SUPPORTED_LANGUAGES` in `i18n.config.js`.
2. Add `fr: { alerts: {...}, email: {...}, messages: {...} }` to `translations`.
3. Add `"fr"` to `language` enum in `Store.model.js`.
4. Add `"fr"` to `allowedLanguages` in `contextBuilder.service.js`.
5. Add `fr: "ALWAYS respond in French."` to `languageInstruction` map.

Never update only one location — all 5 must be updated together.

---

## 8. ADDING NEW ALERT TYPES

When a new alert type is added to `alertEngine.service.js`:

1. Add the new type key to `t.alerts` in ALL languages.
2. Add the corresponding message function to `t.messages` in ALL languages.
3. Never add a translation to only one language — maintain full parity.

## 9. I18N CHECKLIST

When implementing or reviewing i18n changes, verify:

- [ ] All 3 languages (es, en, ca) have identical keys — no missing translations.
- [ ] All message functions receive correct params and handle cents conversion.
- [ ] `getT("invalid_lang")` returns Spanish (default fallback).
- [ ] `getT(undefined)` returns Spanish (null safety).
- [ ] Email subject uses `t.alerts[alert.type]` — not hardcoded string.
- [ ] Email body uses `t.messages` functions — not string concatenation.
- [ ] `store.language` is fetched from DB before calling `getT()`.
- [ ] No translated strings exist outside `i18n.config.js`.

## 10. CODING & LINGUISTIC STANDARDS

- **Code:** English. **Comments:** Spanish (explicar decisiones de traducción y conversión de céntimos).
- **ESM syntax:** `import/export`. Always include `.js` extension.
- **File:** `src/config/i18n.config.js` — single file, no splitting by language or namespace.
- **Anti-patterns:**
  - NO external i18n libraries.
  - NO hardcoded translated strings outside `i18n.config.js`.
  - NO string concatenation for translatable messages.
  - NO translating server logs or thrown error messages.
  - NO partial language support — all keys must exist in all languages.
